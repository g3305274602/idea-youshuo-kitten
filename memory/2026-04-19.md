# 2026-04-19 Cloudflare Pages 建置

- **問題**：CF 在倉庫根執行 `npm run build`；先前 `package.json` 在子目錄會 `ENOENT`；子目錄建置時曾出現 **`tsc: Permission denied`（126）**。
- **目錄**：前端與 `spacetimedb/` 已**扁平化到倉庫根**（不再使用 `youshuo/` 一層）。本機若殘留空資料夾 `youshuo/`（僅鎖定的 `node_modules`），請先關閉 `npm run dev` 後手動刪除該資料夾。
- **建置**：`package.json` 的 `build` 為 **`npm exec tsc -- -b && vite build`**（勿用 `npx tsc`，會誤裝 npm 上名為 `tsc` 的錯誤套件）；並解決 Linux 上直接執行 `tsc` 腳本偶發 **Permission denied**。
- **Cloudflare**：**Build output directory** 設 **`dist`**；**Build command** 用 **`npm run build`**；環境變數 **`VITE_SPACETIMEDB_HOST`**（`wss://maincloud.spacetimedb.com`）、**`VITE_SPACETIMEDB_DB_NAME`**（`idea-jd2zx`）設在 **Production** 的 **Build** 環境（Vite 在建置時內嵌）。未設時，**正式 bundle** 已改為預設連 **maincloud**，不再誤用 `ws://localhost:3000`。
- **線上仍連 localhost**：多為舊版 JS 快取；強制重新整理或清 CDN 快取後再試；若曾用 localhost 登入，請刪除 `localStorage` 裡舊的 `ws://localhost:3000/.../auth_token`（或整站清除），避免 token 與新主機鍵不一致。
- **`insertBefore` / NotFoundError`**：`AnimatePresence mode="wait"` 下第三分支為純 `<div>` 且無 `key`，易與 Motion 退場協調衝突；已改為 **`motion.div`（`key="mailbox-empty"`）** 並為信件詳情 **`motion.div` 補上 `exit`**。`SPACETIME_TOKEN_KEY` 與 `main.tsx` 同樣在 production 預設 **maincloud**。若仍發生，檢查 Cloudflare **Rocket Loader** 等是否改寫 DOM，可關閉後再試。

## 公開廣場（SpacetimeDB + 前端）

- **表**：`scheduled_message.exchangeLog`（雙方往來摘錄字串）；`square_post` 含 `repliesPublic`、`includeThreadInSnapshot` 與合併後 `snapshotContent`；`square_reaction`（`kind`: `up`／`mid`／`down`，每人每貼一筆，可改或 `none` 清除）；`square_comment`、`square_favorite`（`public: true` 以利 codegen；**前端僅訂閱** `squareFavorite.where(collectorIdentity.eq(identity))`，勿訂閱全表）。
- **Reducers**：`publish_to_square`（`repliesPublic`、`includeThreadInSnapshot`）、`send_scheduled_message`／`update_scheduled_message`（同上＋`squareIncludeThread`）、`append_letter_exchange`、`set_square_reaction`、`unpublish_from_square`、`add_square_comment`、`favorite_square_post`、`unfavorite_square_post`；刪原信時連帶刪廣場貼與反應／留言（**不刪收藏快照**）。
- **前端**：撰寫／編輯以卡片選「僅主文／主文＋往來」與「不開放留言／開放留言」；信件詳情有「雙方往來摘錄」輸入；「公開到廣場」彈窗兩段式選項；廣場詳情為讚／中立／踩與收藏，留言僅在 `repliesPublic` 時。
- **部署**：修改模組後請執行 `npm run spacetime:publish`（或 local）並 `npm run spacetime:generate`。
- **遷移**：`scheduled_message.exchangeLog` 須有預設 `''`、`square_post.includeThreadInSnapshot` 預設 `false`，否則舊列可能無法載入、前端收件／寄件列表會像「全沒了」。若曾整庫重建，資料本來就不在雲端，無法由此還原。
- **瀏覽器 `RangeError: Tried to read … byte(s) … remain`**：幾乎都是 **雲端模組與本機 `npm run spacetime:generate` 產物不同步**。處理：依序 `spacetime:generate` → `spacetime:publish`（對應 HOST/DB）→ 硬重新整理；`main.tsx` 已對此錯誤在 console 印出中文提示。
- **maincloud 發佈失敗（manual migration / default value annotation）**：結構變更過大時，一般 `spacetime:publish` 會中止；需 **`npm run spacetime:publish:break`**（等同 `spacetime publish idea-jd2zx … --delete-data=on-conflict`），會**清空該庫模組與資料**後再發佈。已用此法將 `idea-jd2zx` 與目前 schema 對齊。
- **`message channel closed`**：多為 **Chrome 擴充功能** 注入腳本所致，與 SpacetimeDB 無關，可略過或用無痕視窗排除。
